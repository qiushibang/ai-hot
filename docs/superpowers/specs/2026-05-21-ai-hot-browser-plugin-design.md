# AI Hot 浏览器插件设计

## 1. 概述

AI Hot v1 是一个面向 Chrome 的新标签页产品，目标是在用户每次打开新标签页时，按平台展示当日 AI 热门项目与内容，并支持基于项目简介进行手动筛选、收藏，以及在用户完成绑定后将选中的内容一键推送到飞书或微信。

v1 采用“两件套”架构：

1. Chrome 新标签页插件：负责展示、筛选、收藏和推送交互。
2. 本地 companion service：负责每日定时采集、清洗、去重、排序、生成中文摘要，并通过 localhost API 向插件提供数据。

该方案不依赖统一云端后端，安装体验参考 TrendRadar 一类“本地部署后可直接使用”的产品形态。

## 2. 产品目标

### 2.1 目标

- 在新标签页中按平台展示 AI 热门内容。
- 让用户在几分钟内筛出值得关注的项目与信息。
- 让用户可以把筛选后的内容先保存到本地收藏箱，再按需推送到飞书或微信。
- 不要求用户自建云端服务器。

### 2.2 非目标

v1 不包含以下能力：

- 用户登录与跨设备同步
- 兴趣画像与智能推荐
- 跨平台统一总榜
- 自动生成长篇日报
- 公共社区分享
- 多浏览器兼容
- 移动端配套
- 云端账号、订阅与统一托管服务

## 3. v1 范围

### 3.1 产品形态

- Chrome 新标签页插件（v1 仅支持 Chrome）
- 本地 companion service

### 3.2 数据平台

- GitHub
- X / Twitter
- YouTube
- 小红书
- Hugging Face

### 3.3 内容与交互

- 每天本地更新 1 次
- 首页按平台分区展示
- 每个平台默认展示 10 条
- 每条内容展示：原文标题、中文摘要、原始链接、平台标识
- 支持手动关键词筛选
- 支持本地收藏箱 / 稍后看
- 支持用户绑定后的一键推送到飞书或微信
- 所有个人数据默认保存在本地

## 4. 用户流程

### 4.1 首次使用

1. 用户安装 Chrome 插件。
2. 用户通过一键安装器安装本地 companion service。
3. companion service 完成运行环境检查、依赖安装、本地端口配置和定时任务初始化。
4. 用户打开新标签页，插件从本地 localhost API 读取当天聚合结果。

### 4.2 日常使用

1. 用户打开新标签页。
2. 首页按平台分区展示当日热门内容。
3. 用户通过包含关键词、排除关键词和平台过滤缩小结果范围。
4. 用户将感兴趣内容加入收藏箱。
5. 用户在设置页绑定飞书 bot 或微信 bot 后，可一键推送已选内容。

## 5. 信息架构与展示结构

### 5.1 首页结构

首页采用“按平台分区”布局：

- GitHub 区块
- X / Twitter 区块
- YouTube 区块
- 小红书区块
- Hugging Face 区块

每个平台默认展示 10 条，不做跨平台混排。

### 5.2 卡片内容

每条卡片包含：

- 原文标题
- 中文摘要（2 到 3 行）
- 平台来源
- 原始链接
- 可选热度标签

### 5.3 展示原则

- 新标签页只读取本地缓存结果，不等待实时采集完成。
- 结果不足的平台显示“今日结果较少”之类的明确状态。
- 不使用其他平台内容自动补位，保持平台区块结构稳定。

## 6. 系统架构

### 6.1 Chrome 新标签页插件

职责：

- 渲染新标签页首页
- 平台分区展示
- 关键词筛选 UI
- 收藏箱 / 稍后看
- 推送按钮与设置入口
- 调用本地 localhost API 获取数据与执行动作

边界：

- 不直接抓取外部平台
- 不执行定时任务
- 不负责摘要生成与重度数据处理

### 6.2 本地 companion service

职责：

- 定时采集各平台内容
- 平台数据标准化
- 去重、清洗、排序
- 生成中文摘要
- 本地缓存写入
- 暴露给插件调用的 localhost API
- 存储本地配置、收藏、bot 配置和推送记录

边界：

- 不负责复杂前端展示
- 不提供公网服务
- 默认仅服务当前用户本机插件

### 6.3 平台采集适配层

每个平台单独一个 adapter：

- GitHubAdapter
- XAdapter
- YouTubeAdapter
- XiaoHongShuAdapter
- HuggingFaceAdapter

统一输出标准结构：

- `platform`
- `title`
- `summary`
- `url`
- `author`
- `publishedAt`
- `popularitySignals`
- `rawTags`
- `sourceId`

该分层保证单个平台异常时，只影响对应 adapter，不影响整体架构。

### 6.4 存储层

v1 采用轻量本地存储：

- 内容缓存：当天聚合结果、历史抓取快照
- 用户数据：关键词、收藏、bot 配置、推送记录

推荐实现：

- SQLite 或等价轻量本地数据库存储内容与结构化用户数据
- JSON 配置文件保存少量本地偏好或安装级配置

### 6.5 本地 API 层

插件与本地 service 通过 localhost HTTP API 通信。

建议接口：

- `GET /api/feed/today`
- `GET /api/feed/platform/:name`
- `GET /api/favorites`
- `POST /api/favorites`
- `DELETE /api/favorites/:id`
- `POST /api/push/feishu`
- `POST /api/push/wechat`
- `GET /api/settings`
- `POST /api/settings`
- `GET /api/status`

API 仅绑定本机，默认不暴露公网访问能力。

## 7. 数据流

### 7.1 每日更新流程

1. 本地 companion service 按计划启动每日任务。
2. 各平台 adapter 拉取候选内容。
3. 系统统一清洗字段并标准化格式。
4. 系统执行平台内去重与基础质量过滤。
5. 系统计算平台内热度分与增长分。
6. 系统生成中文摘要。
7. 系统写入本地缓存。
8. 插件在用户打开新标签页时读取当天结果并渲染。

### 7.2 前后台分离原则

- 采集、清洗、排序、摘要生成均在后台执行。
- 新标签页只做读取与展示。
- 展示速度不依赖实时抓取速度。

## 8. 排序、去重与筛选规则

### 8.1 排序规则

用户已确认 v1 采用“绝对热度 + 增长速度”的混合策略。

平台内排序分由以下部分组成：

- 热度分：stars、点赞、转发、评论、播放等绝对指标
- 增长分：近 24 小时或近一个采集周期内的增长表现
- 时间衰减：过旧内容自然下沉

v1 仅做平台内排序，不做跨平台统一精确总分。

### 8.2 去重规则

v1 不做跨平台强合并卡片，只做轻去重：

1. 平台内去重
   - 相同 URL
   - 相同 repo / 视频 ID / 帖子 ID / 资源 ID
2. 弱关联标记
   - 如果不同平台明显指向同一项目，可标记为“相关项目”，但仍保留独立卡片

### 8.3 筛选规则

v1 支持：

- 包含关键词
- 排除关键词
- 平台过滤

筛选匹配字段：

- 原文标题
- 中文摘要
- 作者名 / 仓库名
- 平台标签字段

不做兴趣画像与个性化排序。

## 9. 中文摘要策略

### 9.1 输出格式

每条卡片展示：

- 1 行原文标题
- 2 到 3 行中文摘要
- 原始链接
- 平台来源

### 9.2 设计原则

- 新标签页是快速浏览场景，摘要必须短。
- 不生成长篇分析，不在首页塞入深度报告。
- 摘要长度应设置上限，避免后台处理过慢。

## 10. 安装与分发

### 10.1 安装流程

v1 安装为“两步但尽量一键”：

1. 安装 Chrome 插件
2. 运行本地 companion service 安装器

安装器职责：

- 检查运行环境
- 安装依赖
- 初始化本地服务
- 配置端口
- 配置每日任务
- 可选配置开机自启

### 10.2 分发目标

- 用户不需要自建云端服务器
- 用户不需要手动部署后端服务到远程主机
- 体验上尽量接近“安装后可直接用”

### 10.3 约束说明

该方案不是“只装浏览器插件”即可完成全部能力，而是“安装插件 + 安装本地 companion service”后即可使用。

## 11. 推送能力

### 11.1 v1 推送路径

1. 用户选中内容
2. 点击加入收藏箱或推送
3. 若未绑定渠道，提示进入设置页配置
4. 若已绑定，可一键推送到飞书 bot 或微信 bot

### 11.2 推送内容格式

每条推送至少包含：

- 平台名
- 原文标题
- 中文摘要
- 原始链接

### 11.3 v1 不做的推送能力

- 自动群发编排
- 多时段自动推送
- 复杂模板编辑器
- 多人协作推送

## 12. 错误处理与状态反馈

### 12.1 插件侧

- 本地服务未启动时，不显示空白页，应明确提示“本地服务未连接”。
- 推送未绑定时，明确提示用户进入设置页配置。
- 推送失败时，显示具体错误信息。

### 12.2 本地服务侧

- 单个平台采集失败时，不应影响其他平台结果生成。
- 采集错误、摘要生成错误、推送错误应有本地可读日志与状态输出。
- 当天无结果的平台应返回明确状态，而不是静默缺失。

## 13. 非功能要求

### 13.1 稳定性

- 新标签页必须快速打开。
- 插件不等待实时抓取完成。
- 单个平台异常不拖垮全局展示。

### 13.2 隐私

- 关键词、收藏、bot 配置和推送记录默认保存在本地。
- v1 不依赖统一账户体系。

### 13.3 可维护性

- 平台 adapter 独立实现。
- 插件层与 service 层解耦。
- 后续若切换到云端聚合模式，前端可保留大部分接口契约。

### 13.4 性能

- 每天更新 1 次。
- 首页读取缓存，不做重计算。
- 摘要生成与排序计算应有可控上限。

## 14. 测试与验收标准

### 14.1 必测场景

#### 安装与启动

- 用户能完成插件安装与本地 companion service 安装。
- 安装完成后，新标签页能正常打开。
- 插件能连接 localhost 服务。
- 本地服务未启动时，插件能明确提示。

#### 数据更新

- 本地服务每天能成功执行一次更新任务。
- 已接入的平台能产出结果。
- 某个平台抓取失败时，不影响其他平台展示。
- 当天没有结果的平台有明确状态提示。

#### 首页展示

- 新标签页按平台分区展示内容。
- 每个平台最多展示 10 条。
- 每条卡片包含原文标题、中文摘要、原始链接和平台标识。
- 页面打开时不因采集任务阻塞。

#### 筛选与收藏

- 包含关键词筛选生效。
- 排除关键词筛选生效。
- 平台筛选生效。
- 收藏箱新增、删除、查看可正常工作。
- 浏览器重启后本地数据仍保留。

#### 推送

- 未绑定飞书或微信时，点击推送会引导配置。
- 已绑定后，可将选中内容成功推送。
- 推送内容包含标题、摘要、链接和平台。
- 推送失败时能看到明确错误信息。

### 14.2 验收标准

v1 完成的最低标准：

- 能在 Chrome 中安装并使用
- 能通过安装器启动本地 companion service
- 首页能稳定展示按平台分区的热门内容
- 支持手动关键词筛选
- 支持本地收藏箱
- 支持绑定后的一键推送到飞书或微信
- 所有个人配置默认保存在本地
- 提供可复现的安装说明与故障排查说明

### 14.3 成功标准

- 用户每天愿意打开新标签页查看一次
- 用户能在几分钟内筛出值得关注的项目
- 用户能把选中的内容快速推送到自己的工作流中
- 用户不需要维护云端服务器

## 15. 建议的验证顺序

1. 本地 companion service 能跑起来
2. 插件能读取本地 API
3. GitHub + Hugging Face 双平台跑通
4. 首页分区展示跑通
5. 关键词筛选跑通
6. 收藏箱跑通
7. 飞书 / 微信推送跑通
8. 再逐个平台补齐 X / YouTube / 小红书

该顺序的目的是先验证产品骨架与两类核心内容源，再逐步扩展到抓取难度更高的平台。

## 16. 结论

AI Hot v1 的正式设计是：

- 以 Chrome 新标签页为唯一入口
- 以“按平台分区”作为首页结构
- 以“原文标题 + 中文摘要 + 原始链接”作为核心信息单元
- 以“手动关键词筛选 + 本地收藏箱 + 已绑定后的一键推送”作为核心交互
- 以“插件 + 本地 companion service”作为正式架构

该设计兼顾了可安装分发、无统一云端依赖、较强可维护性和足够清晰的 v1 范围，适合作为下一阶段实现计划的基础。
